# دليل مطوري الواجهات الأمامية ومنصة Zalo Marketplace Smart API 🚀

مرحباً بك في الدليل المرجعي الكامل لواجهات برمجة التطبيقات (APIs) الخاصة بمنصة **Zalo Marketplace Smart**. تم تصميم وتطوير هذه الواجهات باستخدام إطار العمل **NestJS** مع قاعدة بيانات **PostgreSQL** وخدمة **Supabase** لتوفير أداء عالٍ، أمان موثوق، وتجربة مزامنة ذكية وسلسة.

## 2. نظام المصادقة وحماية الطلبات

تعتمد المنصة على معيار **JWT (JSON Web Tokens)** للتحقق من هوية المستخدمين وحماية الموارد الحساسة.

### 🔒 ترويسة الطلب الإلزامية لعمليات الحماية (Authorization Header)
لجميع الطلبات التي تتطلب صلاحيات، يجب تضمين رمز الـ JWT الذي تحصل عليه بعد تسجيل الدخول بالشكل التالي:
```http
Authorization: Bearer <JWT_TOKEN>
```

## 3. مجموعة الـ APIs الأساسية وطريقة استخدامها

### أ. إدارة المنتجات والسلع
تتيح هذه المجموعة جلب وعرض وتعديل المنتجات المعروضة في السوق.

#### 1. جلب قائمة المنتجات (GET)
* **المسار:** `/products`
* **الصلاحيات:** عام (Public)
* **الاستجابة الناجحة (200 OK):**
```json
[
  {
    "id": 1001,
    "name": "سماعات أنكر اللاسلكية Soundcore X",
    "description": "سماعات أذن عازلة للضوضاء تدعم البلوتوث 5.3 مع بطارية تدوم 40 ساعة.",
    "price": 7900,
    "stock": 25,
    "storeId": 5,
    "imageUrl": "/assets/products/anker_soundcore.jpg",
    "created_at": "2026-06-30T14:20:00Z"
  }
]
```

#### 2. إنشاء منتج جديد (POST)
* **المسار:** `/products`
* **الصلاحيات:** مقتصر على التجار والمسؤولين (`MERCHANT`, `ADMIN`)
* **بيانات الطلب (Payload):**
```json
{
  "name": "هاتف ذكي ريلمي C67",
  "description": "ذاكرة 256 جيجابايت مع كاميرا 108 ميجابكسل فائقة الوضوح.",
  "price": 31500,
  "stock": 10,
  "storeId": 5,
  "imageUrl": "/assets/products/realme_c67.jpg"
}
```

### ب. إنشاء وتتبع الطلبات
تستخدم هذه الواجهة للتكامل المباشر عند قيام العميل بإجراء عملية الدفع (Checkout). وهي مدعومة بنظام تحقق صارم من البيانات (Validation DTO) مدعوم بـ `class-validator` لتأكيد صحة الكميات والأسعار.

#### 1. إنشاء طلب شراء جديد (POST)
* **المسار:** `/orders`
* **الصلاحيات:** مقتصر على الزبائن المسجلين (`CUSTOMER`)
* **بيانات الطلب المفصلة (Validation Schema):**
```json
{
  "items": [
    {
      "productId": 1001,
      "productName": "سماعات أنكر اللاسلكية Soundcore X",
      "price": 7900,
      "quantity": 2
    }
  ],
  "address": "حي المستقبل، مبنى 12، الطابق 2",
  "wilaya": "الجزائر",
  "commune": "المرسى",
  "paymentMethod": "COD",
  "phone": "0770123456"
}
```

#### 📌 تفاصيل التحقق من حقول الطلب (Validation Rules):
1. **`items`**: مصفوفة إلزامية تحتوي على السلع؛ يتم فحصها بشكل عميق للتأكد من خلوها من أي حقول ناقصة.
2. **`productId`**: يجب أن يكون رقماً صحيحاً وموجباً وصالحاً بقاعدة البيانات.
3. **`price`**: السعر الأحادي، يجب ألا يقل عن 0.
4. **`quantity`**: الكمية المطلوبة، يجب أن تكون عدداً صحيحاً لا يقل عن قطعة واحدة (1).
5. **`paymentMethod`**: يقبل فقط القيم المتفق عليها: `COD` (الدفع عند الاستلام)، `BARIDIMOB` (بريدي موب)، أو `CCP`.

### ج. تحليل الأداء والإنذارات الاستباقية
هذا الـ API متصل بنظام **Telemetry Logger** الذكي لمراقبة صحة المنصة لحظة بلحظة.

#### 1. إرسال سجلات الأداء والأخطاء (POST)
* **المسار:** `/analytics/telemetry`
* **الصلاحيات:** عام (Public)
* **الاستجابة:** يخزن السجلات لتدقيقها وحساب جودة الخدمة.

#### 2. إنذار تخطي نسبة الأخطاء المسموحة (POST)
* **المسار:** `/analytics/alerts/threshold-breach`
* **الوصف:** يتم تفعيله تلقائياً من الواجهات الأمامية إذا تجاوزت نسبة الأخطاء غير المعالجة 2% من إجمالي الطلبات في غضون ساعة واحدة.
* **الpayload المرسل:**
```json
{
  "alertType": "CRITICAL_EXCEPTION_RATE_BREACH",
  "errorRate": "2.15%",
  "totalRequests": 45,
  "totalErrors": 3,
  "timestamp": "2026-07-01T04:12:00Z",
  "details": "تجاوزت نسبة الأخطاء غير المعالجة في المتصفحات حاجز الـ 2% المسموح به في اتفاقية مستوى الخدمة (SLA)."
}
```
* **الإجراء التلقائي:** يقوم الخادم بإرسال إخطار فوري لقناة Slack الخاصة بفريق الصيانة والعمليات أو البريد الإلكتروني.

## 4. جسر الاتصال والتعافي التلقائي (Exponential Backoff with Jitter)

عند استخدام جسر الاتصال `nestjs-bridge.js` لإرسال الطلبات، تم دمج آلية إعادة محاولة ذكية تحمي الواجهة الخلفية من الانهيار في فترات الحمل العالي وتضمن في نفس الوقت سلاسة التجربة للمستخدم:

1. **الانتظار المتضاعف (Exponential Delay):** يتم ضرب وقت الانتظار الأساسي بالأس `2` عند كل محاولة فشل متتالية ليتسع النطاق الزمني التدريجي.
2. **عشوائية الـ Jitter:** يُضاف رقم عشوائي (بين 0 و 300ms) لمنع تزامن طلبات المئات من المستخدمين في نفس اللحظة (Thundering Herd Problem).
3. **الفلترة الذكية:** لا تتم إعادة المحاولة للأخطاء المنطقية من جهة العميل (مثل `400 Bad Request` أو `401 Unauthorized`)، بل يُكتفى بالمعالجة الفورية لإخطار المستخدم بالخطأ الحقيقي.

## 5. محرك المزامنة أوفلاين (Offline-to-Online Sync Engine)

في حال انقطاع الشبكة أثناء تجول المستخدم، يتم تحويل جميع التعديلات والإضافات تلقائياً إلى **طابور العمليات المعلقة (Offline Queue)** المخزن بأمان في الـ LocalStorage. وعند استعادة الشبكة، يعمل محرك المزامنة وفق القواعد التالية:

1. **حل صراع الهوية (Relational Integrity Resolution):** يتم استبدال المعرفات المؤقتة التي تم توليدها أوفلاين بالمعرفات الحقيقية المرتجعة من خادم PostgreSQL فور توليدها، مما يمنع انقطاع العلاقات بين الجداول (مثل ربط السلعة بطلب جديد).
2. **قاعدة الكتابة للأحدث (Last-Write-Wins):** قبل تطبيق أي تعديل، يتم فحص طابع التاريخ والوقت لتفادي مسح أو استبدال بيانات أحدث تم إضافتها مسبقاً من جهاز آخر.
3. **المزامنة الدورية الهادئة:** يقوم المحرك بفحص الطابور تلقائياً فور تحسس عودة الشبكة (`window.online`) ويجري فحصاً دورياً كإجراء احترازي كل 15 ثانية دون أي تشتيت للمستخدم.
